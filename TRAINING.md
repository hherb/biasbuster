# TRAINING.md

This guide provides step-by-step instructions for fine-tuning large language models (LLMs) to detect bias in biomedical research abstracts using the BiasBuster dataset.

## Overview

The training process uses Low-Rank Adaptation (LoRA) to efficiently fine-tune pre-trained LLMs on bias detection tasks. This approach modifies only a small subset of the model's parameters while keeping the base model frozen, making training feasible on consumer hardware.

### Supported Models

- **Qwen3.5-27B**: Strong performance on reasoning tasks, good for structured analysis
- **OLMo-3.1-32B-Instruct**: Open-source model with good generalization capabilities

### Training Pipeline

1. **Data Preparation** (via `export.py`) - Convert annotations to training format
2. **LoRA Fine-tuning** (via `train_lora.py`) - Train the adapter
3. **Model Merging** (via `merge_adapter.py`) - Combine adapter with base model
4. **Deployment** (optional) - Export to Ollama for inference

## Prerequisites

### Hardware Requirements

- **GPU**: At least 24GB VRAM (for 27B parameter models)
- **RAM**: 64GB+ system RAM
- **Storage**: 100GB+ free space for models and checkpoints

### Software Requirements

- **Docker**: For NGC PyTorch container
- **NVIDIA GPU drivers**: Compatible with CUDA 12.1+
- **Python dependencies**: Installed automatically in container

### Data Requirements

- Training data: `dataset/export/alpaca/train.jsonl`
- Validation data: `dataset/export/alpaca/val.jsonl`
- Generated by running `uv run python pipeline.py --stage export`

## Step 1: Data Preparation

Before training, ensure your dataset is properly formatted:

```bash
# Run the full pipeline to generate training data
uv run python pipeline.py --stage all

# Or just the export stage if data collection is already done
uv run python pipeline.py --stage export
```

This creates alpaca-format JSONL files with:
- **system**: Task instructions and bias assessment framework
- **instruction**: The clinical trial abstract to analyze
- **output**: Structured bias assessment with reasoning chains

## Step 2: Model Selection and Configuration

### Choosing a Base Model

**Qwen3.5-27B** (Recommended for production):
- Excellent at structured reasoning and following complex instructions
- Strong performance on scientific text analysis
- Good at generating verification steps with specific database citations

**OLMo-3.1-32B-Instruct** (Alternative):
- Open-source model with permissive licensing
- Good generalization to unseen bias patterns
- May require more training data for optimal performance

### Key Hyperparameters

#### LoRA Parameters

```python
# From training/configs.py
lora_r: int = 16          # Rank of the LoRA adapter
lora_alpha: int = 32      # Scaling factor (alpha/r = 2.0)
lora_dropout: float = 0.05  # Dropout probability in LoRA layers
```

**Why these values?**
- `r=16`: Balances parameter efficiency with expressiveness. Higher values (32, 64) increase capacity but require more memory
- `alpha=32`: With r=16, gives alpha/r=2.0, which is optimal for most tasks
- `dropout=0.05`: Prevents overfitting while maintaining training stability

#### Training Parameters

```python
num_train_epochs: int = 3              # Full passes through training data
per_device_train_batch_size: int = 1   # Samples per GPU per step
gradient_accumulation_steps: int = 4   # Effective batch size = 4
learning_rate: float = 2e-4           # Learning rate (2e-4 is stable for LoRA)
lr_scheduler_type: str = "cosine"     # Learning rate schedule
warmup_ratio: float = 0.1             # 10% of steps for warmup
max_seq_length: int = 4096            # Maximum sequence length
```

**Why these values?**
- `epochs=3`: Sufficient for convergence without overfitting on ~1000 examples
- `batch_size=1` with `grad_accum=4`: Effective batch size of 4 fits in memory
- `lr=2e-4`: Standard for LoRA fine-tuning, stable and effective
- `cosine` scheduler: Smooth learning rate decay prevents plateauing
- `warmup=0.1`: Prevents early training instability
- `max_seq_length=4096`: Accommodates long abstracts with reasoning

#### Target Modules

```python
target_modules = [
    "q_proj", "k_proj", "v_proj", "o_proj",    # Attention layers
    "gate_proj", "up_proj", "down_proj",      # MLP layers
]
```

**Why these modules?**
- Covers all major transformer components
- `q_proj`, `k_proj`, `v_proj`, `o_proj`: Attention mechanism adaptation
- `gate_proj`, `up_proj`, `down_proj`: Feed-forward network adaptation
- Excludes output layer to preserve general language capabilities

## Step 3: Training Execution

### Basic Training

```bash
# Train Qwen3.5-27B from scratch
./run_training.sh qwen3.5-27b

# Train OLMo-3.1-32B
./run_training.sh olmo-3.1-32b
```

### Resuming Training

```bash
# Resume from latest checkpoint
./run_training.sh qwen3.5-27b --resume
```

### Smoke Testing

```bash
# Quick test with only 5 training steps
./run_training.sh qwen3.5-27b --max-steps 5
```

### What Happens During Training

1. **Model Loading**: Base model loaded in bfloat16 precision with device mapping
2. **LoRA Application**: Adapter layers added to target modules
3. **Data Loading**: Alpaca JSONL converted to chat template format
4. **Training Loop**: Supervised fine-tuning with gradient accumulation
5. **Checkpointing**: Automatic saving every 50 steps
6. **Validation**: Loss evaluation on validation set (if available)

### Monitoring Training

The training script logs:
- Trainable parameters percentage (should be ~0.5-1%)
- Training loss every 10 steps
- Validation loss every 50 steps (if validation data provided)
- Checkpoint saving progress

Expected training time: 2-4 hours on A100/H100 GPUs

## Step 4: Model Merging

After training, merge the LoRA adapter into the base model:

```bash
# Merge Qwen adapter
./run_merge.sh qwen3.5-27b

# Merge OLMo adapter
./run_merge.sh olmo-3.1-32b
```

**Why merge?**
- LoRA adapters are inference-inefficient
- Merged models can be deployed anywhere
- Enables quantization and optimization
- Removes dependency on PEFT library at inference time

## Step 5: Deployment to Ollama (Optional)

For local inference, export the merged model to Ollama:

### Full Precision (Recommended)

```bash
# Export Qwen model
bash training/export_to_ollama.sh training_output/qwen3.5-27b-merged qwen3.5-27b-biasbuster

# Export OLMo model
bash training/export_to_ollama.sh training_output/olmo-3.1-32b-merged olmo-3.1-32b-biasbuster
```

### Quantized (Smaller, Faster)

```bash
# Export with Q4_K_M quantization (good balance of size/speed/quality)
bash training/export_to_ollama.sh training_output/qwen3.5-27b-merged qwen3.5-27b-biasbuster --gguf Q4_K_M
```

**Quantization options:**
- `Q4_K_M`: 4-bit, good quality, ~7GB for 27B model
- `Q3_K_L`: 3-bit, smaller, ~5GB, slight quality loss
- `Q2_K`: 2-bit, minimal size, ~3.5GB, significant quality loss

## Step 6: Evaluation and Iteration

### Evaluating Model Performance

```bash
# Run evaluation against human labels
uv run python pipeline.py --stage compare

# Compare multiple models
uv run python pipeline.py --stage compare --models qwen3.5-27b-biasbuster,olmo-3.1-32b-biasbuster
```

### Metrics to Monitor

- **F1 Score**: Per-domain bias detection accuracy
- **Cohen's Kappa**: Agreement with human annotators
- **McNemar's Test**: Statistical significance between models
- **Calibration**: Alignment between confidence and accuracy

### Hyperparameter Tuning

If performance is suboptimal:

1. **Increase LoRA rank**: Try `r=32` for more capacity
2. **Adjust learning rate**: Try `1e-4` or `3e-4`
3. **More epochs**: Try 4-5 epochs if validation loss still decreasing
4. **Larger batch size**: Increase `gradient_accumulation_steps` if memory allows

### Data Iteration

If model shows systematic biases:
1. Review false positives/negatives in evaluation output
2. Add more diverse training examples
3. Refine annotation guidelines
4. Re-run pipeline with improved data

## Troubleshooting

### Out of Memory Errors

- Reduce `per_device_train_batch_size` to 1
- Increase `gradient_accumulation_steps` to 8
- Reduce `max_seq_length` to 2048
- Use gradient checkpointing (enabled by default)

### Poor Training Convergence

- Check data quality: Ensure annotations are consistent
- Verify data format: Alpaca JSONL should have system/instruction/output fields
- Increase `warmup_ratio` to 0.2
- Reduce learning rate to 1e-4

### Validation Loss Not Improving

- Model may be overfitting: Reduce epochs or add regularization
- Data may be insufficient: Collect more training examples
- Annotations may be noisy: Review inter-annotator agreement

### Checkpoint Resume Issues

- Ensure output directory structure is intact
- Check that `resume` flag is used consistently
- Verify HuggingFace cache is accessible in container

```python
MODEL_PRESETS = {
    "your-model": "org/YourModel-Size",
    # ... existing models
}
```

### Custom LoRA Configuration

Modify `LoRATrainingConfig` for specific needs:

```python
# For larger models or more capacity
lora_r: int = 32
lora_alpha: int = 64

# For stability with smaller datasets
lora_dropout: float = 0.1
warmup_ratio: float = 0.2
```

### Multi-GPU Training

The current setup uses `device_map="auto"` for single-node multi-GPU. For distributed training, modify the training arguments to use `Trainer` instead of `SFTTrainer` with appropriate distributed settings.

## Performance Expectations

### Training Metrics
- **Final train loss**: 0.1-0.3 (depending on data quality)
- **Validation loss**: Should decrease steadily and stabilize
- **Trainable parameters**: ~0.5-1% of total model parameters

### Inference Performance
- **Latency**: 10-30 seconds per abstract (depending on quantization)
- **Accuracy**: 80-90% agreement with human experts (domain-dependent)
- **Calibration**: Well-calibrated confidence scores for bias severity

### Model Size
- **LoRA adapter**: ~50-100MB
- **Merged model**: ~50GB (bfloat16)
- **Quantized GGUF**: 7-15GB depending on quantization level
