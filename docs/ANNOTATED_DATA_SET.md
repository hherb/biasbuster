# Annotated Data Set

LLM-generated bias annotations on the seed data set, stored in the
`annotations` table of `dataset/biasbuster.db`.  These annotations are
the training signal for fine-tuning bias detection models.

## How it was created

### Prerequisites

The annotation stage requires a populated seed database (see
[SEED_DATA_SET.md](SEED_DATA_SET.md)).  Annotations are generated by
sending each paper's abstract and metadata to an LLM with a structured
bias assessment prompt.

### Pipeline

```bash
# Annotate with the default model (Anthropic Claude)
uv run python pipeline.py --stage annotate

# Annotate with DeepSeek reasoner
uv run python pipeline.py --stage annotate --models deepseek

# Annotate with multiple models (results stored side by side)
uv run python pipeline.py --stage annotate --models anthropic,deepseek
```

Each annotation is saved to the database **incrementally** via an
`on_result` callback as it completes — annotations survive mid-run
crashes.  The pipeline automatically resumes from where it left off:
already-annotated PMIDs (per model) are skipped.

### Configuration

Key settings in `config.py` (copy from `config.example.py`):

| Setting | Default | Purpose |
|---------|---------|---------|
| `annotation_model` | `claude-sonnet-4-6` | Anthropic model for annotation |
| `annotation_max_tokens` | `4000` | Max output tokens (Claude) |
| `deepseek_model` | `deepseek-reasoner` | DeepSeek model (R1 with CoT) |
| `deepseek_max_tokens` | `16000` | Max output tokens (includes CoT reasoning) |
| `annotation_concurrency` | `3` | Parallel API requests |
| `annotation_delay` | `1.0` | Seconds between requests |
| `annotation_max_per_source` | `{...}` | Per-source annotation limits |

**Important:** For `deepseek-reasoner`, `max_tokens` caps the **total
output including chain-of-thought reasoning**, not just the visible JSON.
The 16K default ensures reasoning-heavy responses are not truncated.

### Prompt

Both annotator backends use the same prompt (`prompts.py:ANNOTATION_SYSTEM_PROMPT`)
as the single source of truth.  The prompt instructs the LLM to:

1. Assess bias across 5 domains (statistical reporting, spin, outcome
   reporting, conflict of interest, methodology)
2. Rate each domain on a 5-level severity scale (none / low / moderate /
   high / critical) using explicit boundary definitions
3. Provide evidence quotes from the abstract
4. Recommend specific verification steps citing real databases (CMS Open
   Payments, ClinicalTrials.gov, ORCID, Europe PMC)
5. Assign an overall severity, bias probability (0.0-1.0), and confidence

The prompt also includes calibration guidance (Cochrane RoB 2 expert
judgments when available), retraction severity floors, and the Boutron
spin taxonomy.

### Data flow

```
papers table (4184 papers)
    │
    ├─ retraction notice filter (is_retraction_notice())
    ├─ excluded flag check (excluded = 0)
    │
    ▼
LLM annotator (Anthropic or DeepSeek)
    │
    ├─ JSON parsing + repair (parse_llm_json())
    ├─ Completeness validation (validate_annotation())
    │   └─ Rejects truncated responses → retried up to max_retries
    │
    ▼
annotations table (pmid + model_name = composite PK)
```

### Annotation structure

Each annotation is a JSON object stored in `annotations.annotation`:

```json
{
  "statistical_reporting": {
    "severity": "none|low|moderate|high|critical",
    "relative_only": true,
    "absolute_reported": false,
    "evidence_quotes": ["..."],
    ...
  },
  "spin": {
    "severity": "...",
    "spin_level": "none|low|moderate|high",
    "conclusion_matches_results": true,
    ...
  },
  "outcome_reporting": {
    "severity": "...",
    "primary_outcome_type": "patient_centred|surrogate|composite|unclear",
    ...
  },
  "conflict_of_interest": {
    "severity": "...",
    "funding_type": "industry|public|mixed|not_reported|unclear",
    ...
  },
  "methodology": {
    "severity": "...",
    "inappropriate_comparator": false,
    "short_follow_up": true,
    ...
  },
  "overall_severity": "none|low|moderate|high|critical",
  "overall_bias_probability": 0.45,
  "reasoning": "Step-by-step reasoning...",
  "recommended_verification_steps": ["Check ClinicalTrials.gov...", ...],
  "confidence": "low|medium|high"
}
```

The `overall_severity`, `overall_bias_probability`, and `confidence`
fields are also stored as indexed columns on the `annotations` table
for efficient querying.

### Quality safeguards

1. **Completeness validation** — `validate_annotation()` in
   `annotators/__init__.py` checks that all 9 required fields are present
   (5 domain dicts with severity + 4 overall fields).  Truncated
   responses are rejected and retried.

2. **Retraction notice filtering** — `is_retraction_notice()` excludes
   bare retraction/withdrawal notices that have no assessable research
   content.

3. **Checkpoint/resume** — The pipeline queries `already_done` PMIDs per
   model before starting, so re-runs skip completed work.

4. **Retraction severity floors (detectable retractions only)** —
   Papers retracted for reasons that ARE detectable from abstract text
   (e.g., statistical errors, flawed analysis, COI) have minimum
   severity floors enforced via the prompt.  Papers retracted for
   reasons NOT detectable from the abstract (e.g., data fabrication,
   fraud, manipulation) are annotated on the abstract's own merits
   with no floor — see below.

### Abstract-detectable vs abstract-undetectable retractions

Not all retraction reasons produce visible bias signals in the abstract
text.  Data fabrication, for example, produces clean-looking abstracts
— the fraud is invisible.  Training a text-based bias detector on these
papers with forced HIGH/CRITICAL severity creates wrong training signal:
the model learns "clean abstract = high bias."

The retraction classifier (`enrichers/retraction_classifier.py`)
classifies each retraction reason as **abstract-detectable** or
**abstract-undetectable**:

| Detectable from abstract | Categories | Training handling |
|--------------------------|-----------|-------------------|
| **No** | fabrication, falsification, fraud, paper mill, misconduct, manipulation, unreliable results, data concerns, data not available, misrepresentation, irreproducible, image concerns, peer review concerns, ai generated, unknown retraction | Annotated WITHOUT retraction context; no severity floor; LLM rates abstract on its own merits |
| **Yes** | statistical errors, analytical errors, flawed analysis, conflict of interest, ethics violation, policy breach | Annotated WITH retraction context and severity floor enforcement |

**For evaluation/harness testing**: abstract-undetectable papers are
excellent test cases.  The agent harness should discover the retraction
via external database checks (Retraction Watch, Crossref) and flag it
accordingly — this tests the full verification pipeline, not just
abstract-level detection.

## Checking alignment with expert assessments

### Quick check

```bash
uv run python expert_rob_alignment_of_annotations.py
```

This compares annotations against Cochrane RoB 2 expert assessments
(the only expert ground truth currently available) and prints a Markdown
report to stdout.  Redirect to a file if desired:

```bash
uv run python expert_rob_alignment_of_annotations.py > alignment_report.md
```

### What the report shows

1. **Coverage** — How many annotations have expert ground truth vs don't,
   broken down by Cochrane RoB level (low / some_concerns / high)

2. **Confusion matrix** — 3x3 table comparing expert vs LLM on a
   collapsed scale (Cochrane's 3 levels mapped to our 5):
   - Cochrane "low" = our {none, low}
   - Cochrane "some_concerns" = our {moderate}
   - Cochrane "high" = our {high, critical}

3. **Weighted Cohen's kappa** — Ordinal agreement statistic with
   Landis & Koch interpretation (0.61-0.80 = substantial, etc.)

4. **Binary detection** — Precision/recall/F1 for detecting any concern
   (moderate+ severity) vs no concern (low)

5. **Directional bias** — Whether the LLM systematically over-flags
   (rates higher than expert) or under-flags (rates lower)

6. **Retraction severity floor compliance** — Do annotated retracted
   papers meet the minimum severity implied by their retraction reason?
   (e.g., data fabrication should be critical, manipulation should be high)

7. **Heuristic suspicion alignment** — Do papers flagged as
   high-suspicion by the effect size auditor/outcome switching heuristics
   get appropriately higher severity ratings?

8. **Confidence calibration** — Is the model's self-reported confidence
   meaningful? High confidence should correlate with clear-cut cases
   (extreme severity), medium with ambiguous cases (moderate)

9. **Per-model breakdown** — If multiple models annotated, shows
   kappa/F1 per model for comparison

10. **Interpretation guide** — Caveats about construct mismatch
    (methodology from full text vs reporting bias from abstract only),
    information asymmetry, and scale compression

### Interpreting results

**Expected behaviour:**
- Kappa in the moderate-to-substantial range (0.4-0.8) is good, given
  that Cochrane assesses methodology from full text while our LLM
  assesses reporting bias from abstracts only
- Some over-flagging on reporting dimensions is expected and may be
  correct (spin and selective statistics are visible in abstracts)
- Under-flagging on methodology is expected (allocation concealment
  and blinding details are rarely in abstracts)
- High-RoB papers should be detected reliably (>80% rated high)
- Low-RoB papers may show more moderate ratings due to reporting
  concerns not captured by Cochrane's methodology focus

**Red flags to investigate:**
- Kappa below 0.3 (poor agreement)
- Strong directional bias (mean difference > |0.3|)
- Low-RoB papers frequently rated high (systematic over-flagging)
- High-RoB papers frequently rated low (model not learning from
  Cochrane calibration data in the prompt)

## Reproducing from scratch

```bash
# 1. Start from seed data (or re-collect)
uv run python seed_export.py import

# 2. Run annotation (takes several hours depending on model/API speed)
uv run python pipeline.py --stage annotate --models deepseek

# 3. Check alignment
uv run python expert_rob_alignment_of_annotations.py

# 4. (Optional) Human review via NiceGUI web tool, then export
uv run python pipeline.py --stage export
```

The annotation stage is idempotent — running it again skips
already-annotated papers and picks up any new or previously failed ones.

## Export to training format

```bash
uv run python pipeline.py --stage export
```

Produces fine-tuning data in two formats with 80/10/10 train/val/test
splits (seed=42 for reproducibility):

```
export/
├── alpaca/
│   ├── train.jsonl    # instruction/input/output with <think> chains
│   ├── val.jsonl
│   └── test.jsonl
└── sharegpt/
    ├── train.jsonl    # multi-turn conversation format
    ├── val.jsonl
    └── test.jsonl
```

The alpaca format includes `<think>` reasoning chains that teach the
model *where* to look (which databases, which flags) and *why* a
severity level was assigned, not just the final rating.
