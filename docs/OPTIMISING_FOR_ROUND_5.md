# Optimising for Round 5: Fixing Severity Calibration

**Date:** 2026-03-27
**Status:** Planning
**Prerequisite:** `docs/ROUND_4.md` (V7/V8 results), `docs/MISTAKES_ROUND_1_AND_FIXES.md`

---

## The Problem

After 4 rounds and 10+ training runs, the pattern is clear:

| What works | What doesn't |
|------------|-------------|
| Binary bias detection (F1: 0.929 → 0.966) | Severity grading (kappa stuck at ~0.095) |
| Calibration error (0.802 → 0.306) | Low/high severity distinction |
| Verification source knowledge (0.460 → 0.517) | Training data distribution match |
| Domain-level bias detection | Per-domain severity calibration |

**Fine-tuning improves detection while degrading calibration.** The
pre-trained baseline (zero-shot) almost perfectly matches the ground truth
severity distribution. Every fine-tuned version distorts it.

---

## Root Cause Analysis

### 1. Training data is 61% moderate

```
Training data:    none=3.4%  low=13.8%  moderate=60.9%  high=19.6%  critical=2.2%
Ground truth:     none=4.1%  low=13.8%  moderate=59.3%  high=19.5%  critical=3.3%
Test set (123):   none=4.1%  low=13.8%  moderate=59.3%  high=19.5%  critical=3.3%
```

The distributions look similar, yet the model over-predicts high (24 GT →
65 V7 predicted) and under-predicts low (17 GT → 1 V7 predicted). Why?

The issue is not the marginal distribution but the **conditional**: the
model learns that most papers have *some* bias concern (true) and then
defaults to moderate-to-high because that's the safe middle ground. The
training signal for distinguishing low from moderate from high is weak
because the `<think>` reasoning and structured response use **identical
patterns** across severity levels — the only difference is the label.

### 2. Training output format is markdown, not JSON

`export.py` builds the training output as:
```
<think>
[reasoning]
</think>

## Statistical Reporting: MODERATE
- Only relative measures reported...

## Spin: HIGH
- Conclusions do not accurately reflect...

## Overall: MODERATE (bias probability: 40%)
```

This format has two problems:
- **Severity labels are embedded in prose headers**, not structured fields.
  The model doesn't get a clear signal that severity is a classification
  task — it looks like free-text generation.
- **Adding JSON instructions to the system prompt** (V8) conflicts with
  every training example showing markdown. The attention-only LoRA cannot
  resolve this contradiction with 0.04% trainable params.

### 3. `<think>` reasoning doesn't discriminate severity levels

The thinking chain template (`build_thinking_chain`) produces similar
reasoning for low and moderate concerns. A study with one minor statistical
omission (LOW) and a study with multiple reporting concerns (MODERATE) get
nearly identical reasoning patterns. The model has no clear signal for
*where the boundary is*.

### 4. Round 1 lesson still applies

From `MISTAKES_ROUND_1_AND_FIXES.md`:
> "The annotation prompt IS the ground truth. If it doesn't define severity
> boundaries, the labels are whatever the LLM felt like on that day."

The annotation prompt now includes severity boundaries (fixed in Round 2).
But the ground truth labels were generated by DeepSeek, which has its own
calibration tendencies. The model is learning DeepSeek's calibration, then
applying it through attention-only LoRA, which can only shift attention
patterns, not fundamentally recalibrate output distributions.

## Strategy for Round 5

### Priority 1: Change training data format to JSON

**What:** Modify `build_structured_response()` in `export.py` to output
the same JSON schema that the scorer expects. The training output becomes:

```
<think>
[reasoning with severity boundary references]
</think>
{
  "statistical_reporting": {"severity": "moderate", "relative_only": true, ...},
  "spin": {"severity": "high", ...},
  ...
  "overall_severity": "moderate",
  "overall_bias_probability": 0.55,
  ...
}
```

**Why:** The model currently sees markdown in training and is asked for
JSON at inference. With attention-only LoRA, the model cannot learn a new
output structure from a system prompt alone — it needs to see the target
format in every training example. The JSON format also gives a cleaner
training signal: severity is a structured field, not prose embedded in a
markdown header.

**Risk:** Low. The `<think>` block remains unchanged (proven to work).
Only the post-`</think>` portion changes from markdown to JSON. The scorer
already handles JSON natively via `_parse_from_json`.

**Implementation:**
1. Add `build_json_response()` to `export.py` that outputs the annotation
   dict as clean JSON (matching `_JSON_SCHEMA` in `prompts.py`)
2. Update `to_alpaca_format()` and `to_sharegpt_format()` to use it
3. Re-export dataset
4. Validate: every training example must end with valid JSON after `</think>`

### Priority 2: Strengthen severity boundary signal in `<think>` chains

**What:** Modify `build_thinking_chain()` to explicitly reference the
severity boundary definitions when justifying each severity assignment.

Current pattern (generic):
```
Statistical reporting has some concerns. Severity: MODERATE.
```

Target pattern (boundary-grounded):
```
Statistical reporting: relative measures only (RRR without ARR or NNT),
no baseline risk reported. Per severity boundaries, "relative measures
only OR selective p-value reporting" maps to MODERATE. Single dimension
concern, no evidence of intentional obfuscation, so not HIGH.
```

**Why:** The model needs to learn the decision boundary, not just the
label. Explicitly stating *why it's MODERATE and not HIGH* teaches the
distinction. This is especially important for the moderate/high boundary
where V7 makes 43 errors.

**Risk:** Medium. More verbose thinking chains may push against token
limits. Monitor average output length and adjust `max_seq_length` if
needed.

**Implementation:**
1. Update `_build_*_reasoning()` functions in `export.py` to include
   boundary citations (import from `prompts.py`)
2. Add "not X because..." negative reasoning for adjacent severities
3. Validate: spot-check 20 examples for boundary coherence

### Priority 3: Add severity-contrastive examples

**What:** For the most confused severity pair (moderate vs high), create
explicit contrastive training examples that highlight the distinction.

**Why:** The confusion matrix shows 43 moderate→high misclassifications
in V7. The model needs paired examples that show: "this is MODERATE
because X; this similar-looking paper is HIGH because Y."

**Options (in order of preference):**
1. **Mine existing data:** Find pairs of papers with similar domain
   profiles but different overall severities. Include both in the same
   training batch with explicit contrast in the reasoning.
2. **Expert review:** Have the human reviewer specifically audit the
   moderate/high boundary cases and add boundary reasoning to annotations.
3. **Synthetic contrast:** Generate paired examples where a small change
   (e.g., adding outcome switching evidence) flips the severity from
   moderate to high.

**Risk:** Medium. Contrastive examples are powerful but can overfit to
specific patterns if the set is too small. Need at least 30-50 pairs.

### Priority 4: Evaluate whether `<think>` blocks help or hurt

**What:** Run a training experiment without `<think>` blocks — JSON only.

**Why:** The `<think>` block consumes 60-70% of output tokens. If severity
calibration is the bottleneck, the model may benefit from focusing entirely
on the structured output rather than splitting its attention between
reasoning and classification.

The baseline (zero-shot) has better calibration with no thinking at all.
This suggests the `<think>` reasoning may be *decorrelating* the model's
severity assessments from its natural calibration.

**Implementation:**
1. Export a variant dataset with `include_thinking=False`
2. Train V9-nothink with same hyperparameters as V7/V8
3. Compare kappa and calibration against V7/V8
4. If kappa improves significantly, the `<think>` blocks need redesign
   (Priority 2) before being re-added

### Priority 5: Consider label smoothing for severity calibration

**What:** Re-add `label_smoothing_factor=0.05` to `_MOE_OVERRIDES`.

**Why:** Label smoothing was removed after Run 8 because it was blamed for
gradient collapse — but the collapse was actually caused by 8e-6 LR (see
`ROUND_3.md` §4.2). At 5e-6 LR with 2 epochs, label smoothing may help
prevent the model from becoming overconfident on modal severity classes
(moderate) at the expense of tail classes (low, high, critical).

**Risk:** Low, since the root cause of Run 8's collapse was LR, not label
smoothing. But should be tested as an ablation, not combined with other
changes.

## What NOT to Do

Based on documented mistakes across all rounds:

1. **Don't change hyperparameters to fix a data problem.** Severity kappa
   is stuck because of training data format and distribution, not because
   LR or rank are wrong. LR=5e-6, rank=32, 2 epochs are proven stable.

2. **Don't add prompt instructions that conflict with training data.**
   V8 added JSON instructions while training data was markdown. The model
   can't resolve this with attention-only LoRA.

3. **Don't fight the model's natural format.** Attention-only LoRA cannot
   learn new structural patterns (Harmony channels, JSON from markdown).
   Instead, make the training data match what you want the model to output.

4. **Don't change more than one thing at a time.** Every round that changed
   multiple variables (Run 8: LR + rank + epochs + label smoothing) made
   it impossible to attribute outcomes to specific changes.

5. **Don't forget to sanity-check raw outputs.** Every round had at least
   one measurement bug (harness discarding output, wrong max_tokens, parser
   matching wrong text). Always `curl` the model and inspect 5 outputs
   before trusting aggregate metrics.

## Proposed Round 5 Plan

### Phase A: Data format fix (Priority 1 + 2)

1. Modify `export.py`: `build_json_response()` replacing
   `build_structured_response()`
2. Update `build_thinking_chain()` with boundary-grounded reasoning
3. Re-export dataset
4. Validate: 100% JSON parse success, spot-check boundary reasoning
5. Train as V9 with unchanged hyperparameters (LR=5e-6, rank=32, 2 epochs)
6. Evaluate V9 vs V7 vs baseline

**Success criteria:** Overall kappa > 0.15, moderate→high errors < 30
(down from 43).

### Phase B: Ablation (Priority 4, only if Phase A kappa < 0.15)

7. Export no-think variant
8. Train V9-nothink with same hyperparameters
9. Compare kappa: if V9-nothink >> V9, the `<think>` design needs rework
10. If V9-nothink ≈ V9, the bottleneck is elsewhere (ground truth quality)

### Phase C: Fine-tuning (Priority 3 + 5, only if Phase A kappa > 0.15)

11. Mine contrastive pairs from moderate/high boundary
12. Add label smoothing (0.05) as ablation
13. Train V10 with contrastive examples + label smoothing
14. Evaluate V10 vs V9

**Target metrics for Round 5 completion:**

| Metric | Current best | Target |
|--------|:---:|:---:|
| Binary F1 | 0.966 (V8) | >= 0.95 (maintain) |
| Severity kappa | 0.098 (baseline) | >= 0.20 |
| Calibration Error | 0.306 (V7) | <= 0.30 |
| Verification Score | 0.517 (V7) | >= 0.50 (maintain) |
| moderate→high errors | 43 (V7) | <= 25 |
